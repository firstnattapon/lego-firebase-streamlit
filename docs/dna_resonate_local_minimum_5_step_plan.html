ตัวชี้วัด: Resonance Spectrum Score = Σnet_w / Σceiling_w ปัจจุบัน: ยังไม่วัด (TBD — ต้องรัน baseline ก่อน) เป้า: 0.90 (สมมติฐาน) ระยะเวลา: 10 วันทำการ
เกณฑ์สำเร็จ:
  1. Global resonance score สูงกว่า baseline greedy เดิมทุก ticker (Δscore ≥ 0 เสมอ)
  2. DNA ที่ tune ได้ decode ผ่าน dna_engine.py เดิมตรงบิตต่อบิต 100%
  3. ปิดฟีเจอร์ใหม่ (num_restarts=1, sa_T0=0) แล้วผลต้องเหมือนโค้ด greedy เดิมทุกประการ (regression guard)
  4. มีหน้า tuning ใช้งานได้จาก repo lego-firebase-streamlit พร้อม dashboard resonance score

2 Problems — ช่องว่างที่ขวางอยู่

P1 — Greedy accept-only, single-champion hill-climb severity 5/5

รับเฉพาะ mutation ที่ดีกว่าเท่านั้น (generate_actions_hybrid_multi_mutation :337-344) ทำให้ไต่ไม่พ้นหลุมท้องถิ่นแรกที่เจอ

Gap: โอกาสพลาด global optimum ต่อ window ~15-25% ของ resonance score

P2 — สำรวจ basin เดียว severity 4/5

Phase 1 (find_best_seed_for_window :242-271) เก็บ base seed ที่ดีที่สุดตัวเดียวจาก 1000 seeds แล้วทิ้ง basin อื่นทั้งหมด

Gap: จำกัดการสำรวจเหลือ 1 ใน 1000 เส้นทางที่เป็นไปได้

P3 — Mutation rate คงที่ 10% severity 3/5

ไม่ปรับตามระยะการวิวัฒน์ (early vs late) จำกัดความหลากหลายของ challenger ในรอบท้าย

P4 — จำนวนรอบ mutation ตายตัว severity 2/5

num_mutations=5 ไม่มีเงื่อนไข convergence/patience — รอบที่ไม่ดีขึ้นก็ยังวนต่อจนครบ

P5 — Tuning อยู่คนละ repo กับ dashboard severity 3/5

Optimizer อยู่ที่ streamlit-example-1 ส่วน dashboard เป้าหมายอยู่ lego-firebase-streamlit — ต้องสลับ repo เพื่อ tune แล้วโหลดผลมาดู

P6 — Tie-order nondeterministic severity 2/5

การเลือก seed ที่ดีที่สุดผูกกับลำดับ thread เสร็จ (as_completed, :262-264, :295-297) รบกวนความน่าเชื่อถือของการวัด A/B

3 Diagnosis — รากของปัญหา

RC1 (→P1) — Acceptance rule เป็น strict '>' เท่านั้น น้ำหนัก w = 0.7

  1. ทำไมติดหลุม? เพราะรับเฉพาะที่ดีกว่า
  2. ทำไมรับเฉพาะดีกว่า? เพราะออกแบบเป็น greedy hill-climb ไม่ใช่ metaheuristic
  3. ทำไมเลือก greedy? เพราะง่าย เร็ว deterministic แต่ไม่ทนต่อ local minimum

หลักฐาน: โค้ด :341 if mutated_net > current_best_net: เป็นเงื่อนไขเดียวที่ตัดสินรับ/ไม่รับ

RC2 (→P1) — ไม่มี temperature/annealing schedule น้ำหนัก w = 0.3

  1. ทำไมไม่มีกลไกลดความเสี่ยงตามเวลา? เพราะไม่เคยออกแบบ SA ไว้ตั้งแต่ต้น

หลักฐาน: ไม่พบตัวแปร temperature/schedule ในไฟล์

RC3 (→P2) — Reduction เป็น single-max ไม่ใช่ top-K น้ำหนัก w = 1.0

  1. ทำไมสำรวจ basin เดียว? เพราะ reduction จาก 1000 seed เหลือ argmax ตัวเดียวทันที ไม่เก็บอันดับรอง

หลักฐาน: โค้ด :262-264 เก็บเฉพาะค่า max สุดท้าย

RC4 (→P3) — mutation_rate เป็นพารามิเตอร์ตัวเดียวคงที่ น้ำหนัก w = 1.0

  1. ทำไมคงที่? เพราะ config เก็บเป็นค่าเดียว ไม่ผูกกับรอบ (round index) ไม่รองรับ schedule

หลักฐาน: load_config() :74-82 กำหนด mutation_rate เป็นตัวเลขเดียว

RC5 (→P4) — Loop control ตายตัว ไม่มี patience counter น้ำหนัก w = 1.0

  1. ทำไมไม่หยุดเมื่อไม่ดีขึ้น? เพราะไม่มีตัวแปรติดตามจำนวนรอบที่ไม่ดีขึ้นติดกัน

หลักฐาน: โค้ด :337 วนลูปตายตัวตามจำนวนรอบที่ config กำหนด

RC6 (→P5) — โปรเจกต์แยก repo ตั้งแต่ต้น น้ำหนัก w = 1.0

  1. ทำไมอยู่คนละที่? เพราะ dashboard repo ออกแบบให้อ่านอย่างเดียวจาก Firebase แต่ optimizer พัฒนาแยกในโปรเจกต์วิจัย

หลักฐาน: lego-firebase-streamlit มีเพียง streamlit_app.py 93 บรรทัด อ่าน Firebase RTDB อย่างเดียว ไม่มีโค้ด optimizer

RC7 (→P6) — Reduce ผลจาก thread pool ไม่ tie-break กับ seed น้ำหนัก w = 1.0

  1. ทำไมไม่ deterministic? เพราะลำดับงานที่เธรดทำเสร็จไม่คงที่ทุกครั้ง (as_completed)

หลักฐาน: โค้ด :262-264, :295-297 ใช้ strict > เทียบทีละผลโดยไม่ tie-break

สมการวัดผล & ประเมินผล

Window resonance score (KPI หลัก)score_w = clamp(net_w / ceiling_w, 0, 1) · ถ้า ceiling_w ≤ ε_c: score_w = 1 เมื่อ net_w ≥ 0 มิฉะนั้น 0สัดส่วนของ wealth ที่ optimizer เก็บได้เทียบเพดาน perfect-foresight ต่อ window
Global Resonance ScoreScore = Σ_w(net_w) / Σ_w(ceiling_w)   target ≥ 0.90คะแนนรวมทุก window ถ่วงน้ำหนักด้วยทุน — headline KPI
Gap-Closed %gap_closed = (Score_upgraded − Score_greedy) / (1 − Score_greedy)   target ≥ 0.15วัดว่ามาตรการใหม่ปิดช่องว่างจาก baseline ไปได้เท่าไหร่
Decode Fidelity Ratefidelity = (จำนวน window ที่ decode_dna(emitted) == champion บิตต่อบิต) / (จำนวน window ทั้งหมด)   target = 1.0ยืนยันว่า DNA ที่ปล่อยออก decode ได้ตรงด้วย dna_engine.py เดิม 100%
Goal attainmentA = G_achieved / G_targetA ≥ 1 คือบรรลุเป้า Global Resonance Score
Progress ratioP = (G_n − G_0) / (G_target − G_0)สัดส่วนระยะทางที่เดินมาแล้วจาก baseline สู่เป้า

🔁 Control loop — เมื่อยังไม่บรรลุเป้าหมาย

สมการควบคุม (error)e_n = G_target − G_n
อัตราการลู่เข้าr_n = |e_n| / |e_(n−1)|

เงื่อนไขหยุด: ถ้า |e_n| ≤ ε = 0.02 → บรรลุเป้าหมาย · gain K = 0.5 · n_max = 5 รอบ · ถ้ายังไม่ถึง → ไต่ escalation ladder:

สถานการณ์แปลว่าวนกลับไป
r_n < 1 (ช่องว่างหด)แผนได้ผล ต้องทำซ้ำStep 5 · เพิ่ม num_restarts หรือปรับ T0/patience ตาม u_(n+1)=K·e_n
r_n ≈ 1 (นิ่ง)น้ำหนักรากปัญหาผิด (RC1 vs RC3)Step 3 · วินิจฉัยใหม่ จัดอันดับ S_A/S_B/S_C ใหม่
r_n > 1 (แย่ลง)ระบุปัญหาผิดStep 2 · ทบทวนหา basin/parameter อื่นที่ยังไม่ระบุ
หลายรอบยังห่างมากเป้าไม่สมจริงStep 1 · ปรับ target 0.90 หรือขยาย horizon ตาม baseline จริง

! ข้อสมมติฐาน

  • current (baseline score) ตั้งเป็น TBD เพราะยังไม่เคยวัด resonance score ของโค้ด greedy ปัจจุบันเป็นตัวเลข — ต้องรัน baseline (num_restarts=1, sa_T0=0) ก่อน
  • target 0.90 เป็นค่าสมมติฐานเบื้องต้น รอปรับหลังเห็น baseline จริงในรอบแรกของ control loop
  • expected_effect (Δ) ของ S_B/S_A/S_C เป็นค่าประมาณจากลักษณะของ intervention ไม่ใช่ผลวัดจริง ใช้จัดลำดับ priority เบื้องต้นเท่านั้น
  • cost (c) หน่วยเป็นชั่วโมงทำงานโดยประมาณของ 1 dev
  • แผนนี้ครอบคลุมเฉพาะการออกแบบ — ยังไม่ implement โค้ด optimizer จริงในรอบนี้ตามที่ตกลงกับผู้ใช้
